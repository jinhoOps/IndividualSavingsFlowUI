# 통합 앱 간 데이터 브리지 가져오기 패턴

## 개요
이 문서는 Step1(가계 흐름 시뮬레이션)의 결과 데이터를 Step2(투자 포트폴리오 설계)로 가져올 때 발생하는 부작용을 방지하기 위한 상태 관리 원칙을 정의합니다.

## 핵심 문제 및 원인
Step1의 데이터를 Step2로 가져오는 과정에서, 가져오기 함수가 활성화된 현재 포트폴리오 상태(state.currentPortfolioId)와 초안 상태(state.draft)를 강제로 초기화하는 문제 패턴이 발견되었습니다. 이렇게 구현할 경우 사용자가 Step2에서 이미 작업 중이던 물리적 계좌 구성, 종목 할당 비중, 설정 내역이 모조리 유실되는 심각한 결함이 발생합니다.

## 상태 보존 및 병합 원칙 (v0.7.0 업데이트)
앱 데이터 연동 기법을 수정하거나 신규 설계할 때는 다음의 데이터 보존 원칙을 반드시 준수해야 합니다.

1. 선택 상태 보존 
데이터를 가져올 때 state.currentPortfolioId 속성은 절대 빈 문자열로 날려버리지 않고 현재 식별자를 그대로 유지해야 합니다. 
2. 객체 선택적 병합 
state.draft 객체를 새로 생성(createEmptyDraft)하지 마십시오. 기존의 시뮬레이션 설정(yield, growth 등)을 보호하며, 브리지에서 제공되는 '월 투자 여력' 데이터만을 반영합니다.
3. 데이터 단위 정합성 (핵심)
브리지 데이터(`totalMonthlyInvestCapacity`)는 원 단위로 수신됩니다. Step 2 UI(만원 단위)와 연동 시 반드시 `IsfUtils.toWon` 및 `toMan`을 사용하여 단위 오차(10,000배)가 발생하지 않도록 철저히 관리해야 합니다.
4. 사용자 여정의 연속성
Step 1에서 '적용'을 누르는 행위가 Step 2의 시뮬레이션 결과에 즉시 반영되도록 브리지 배너를 통해 시각적 피드백을 제공합니다.

---
*구식 패턴 아카이브:*
- (v0.6.0 이전) Step 1의 투자 항목을 Step 2의 계좌로 매핑하던 복잡한 로직은 Step 3 구현 시 재도입될 예정이며, 현재 Step 2에서는 단순 투자 여력 연동으로 간소화되었습니다.
