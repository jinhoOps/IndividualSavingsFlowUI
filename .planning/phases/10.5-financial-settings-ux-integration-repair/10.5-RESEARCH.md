## User Constraints (from CONTEXT.md)

### Locked Decisions

Phase 10.5 repairs the Step 1 financial settings UX by replacing the split household-budget/modal contradiction with one integrated `재무설정 상세` flow. The default Step 1 screen stays summary-first and low-fatigue; dense editing moves into the existing financial modal pattern.

This phase must not restore a separate `#householdBudgetPanel`, must not introduce nested modals, and must not replace the existing sanitizer, persistence, draft save/cancel, or Sankey architecture.

**Important override from discussion:** The original SPEC includes `부부합산`, `본인 설정`, and `배우자 설정` in the ordered detail flow. The user explicitly changed that requirement during discussion: all couple/household-status UI features are removed from Phase 10.5 and deferred to a future phase with a separate UI-SPEC. Planning must reconcile SPEC/test criteria with this user decision before implementation.

**In scope (from SPEC.md, adjusted by discussion):**
- Compact default Step 1 financial summary with a single next action into detail editing.
- One `재무설정 상세` modal/equivalent detail surface for income, living expenses, investment, savings, and automatic-savings result.
- Living-expense child details for variable target, actual, remaining, status, and month-end projection.
- Copy, spacing, visual cleanup, and UX simplification needed to satisfy the Phase 10.5 UI audit threshold.
- Tests or verification updates that align the current contract with integrated financial settings and the removal of Phase 10.5 couple UI.

**Out of scope (from SPEC.md plus discussion):**
- Restoring a separate `#householdBudgetPanel` default-screen panel.
- Couple/household-status UI: `부부합산`, `본인 설정`, `배우자 설정`, spouse income editing, and household mode surfaces.
- Phase 11 pasted bank/card notification capture.
- Phase 12 dual shared-data merge.
- Phase 14 real-estate affordability planning.
- Replacing Step 1 persistence, storage, sanitizer, or Sankey architecture.

Implementation decisions:
- D-01: Add one clear default-screen action into `재무설정 상세`.
- D-02: Existing financial summary card clicks should also open the same integrated `재무설정 상세` flow, not separate category-only modal experiences.
- D-03: The default screen remains light. No default-screen budget panel, dense editor, variable-budget rows, or household-status board should be rendered.
- D-04: Use a tab/step-based detail structure, but keep an always-visible summary rail so the whole monthly cash-flow picture remains visible.
- D-05: Phase 10.5 order after removing couple UI is: `월 수입`, `월 생활비`, `투자`, `저축`, `결과/자동 저축`.
- D-06: The summary rail shows the key five values by default: `월 수입`, `월 생활비`, `월 투자`, `자동 저축`, and status badge.
- D-07: The summary rail stays simple unless a problem exists. Show saving/investment over-budget warning and adjustment action only when `저축 + 투자` exceeds available income.
- D-08: `월 생활비` uses compact child rows: default state shows only row summary, and selecting a row expands target, actual, remaining amount, status, `월말 예상`, and helper copy.
- D-09: Variable expense actual/target controls belong under `월 생활비`. Fixed expenses do not gain actual-spending controls in this phase.
- D-10: The projection helper copy must remain visible where month-end projection appears: `현재 사용 속도를 단순 환산한 참고값입니다.`
- D-11: `자동 저축` appears both in the always-visible summary rail and in the final result tab.
- D-12: `자동 저축` is a derived result, not a manual input equivalent to income, expense, investment, or savings rows.
- D-13: The result tab should connect `자동 저축` to existing savings item/account details without opening a nested modal.
- D-14: Nested modals are prohibited. If the user wants savings account details from the result tab, navigate within the same `재무설정 상세` modal to the `저축` tab.
- D-15: `자동 저축 보정` means handling cases where `저축 + 투자` exceeds available income. The app must not silently mutate durable state before user choice.
- D-16: When `저축 + 투자` exceeds total income/available flow, present three adjustment choices: `투자 먼저 줄이기`, `저축 먼저 줄이기`, or `저축/투자 비율 유지해서 같이 줄이기`.
- D-17: Save proceeds only after the user chooses an adjustment basis or resolves the excess. The selected adjustment is applied inside the existing draft/save boundary.
- D-18: Each tab should start with one large, simple input or summary value. Detailed line items are revealed only when selected.
- D-19: Preserve Phase 09's selected-item editing pattern: do not show all inputs for every item at once.
- D-20: Edits remain draft-only until `재무설정 저장`; `편집 취소` and close discard changes.
- D-21: User convenience is the top design priority: input should feel intuitive, simple, and low-fatigue.
- D-22: Avoid crowded tables, nested modal flows, many always-open inputs, and mobile layouts that require horizontal scanning.
- D-23: Keep the strict UI review gate: overall >= 18/24, and Copywriting, Visuals, Spacing, and Experience Design each >= 3/4.
- D-24: The UI must use plain, neutral copy. No guilt, blame, partner-responsibility framing, or financial/legal/tax/lending advice language.
- D-25: Verification must cover both the core user flow and existing regression contracts.
- D-26: User-flow tests should cover opening `재무설정 상세`, navigating tabs, editing big numbers, expanding variable expense rows, choosing an excess adjustment, save/cancel behavior, and mobile overflow.
- D-27: Regression tests must preserve: no `#householdBudgetPanel`, `#summaryCards` before Sankey, sanitizer/persistence/Sankey behavior, no repeated-save duplicates, and no couple UI in Phase 10.5.

## Architecture Patterns

### System Architecture Diagram

```text
User clicks default CTA or summary card
  -> financial-summary-renderer button / CTA event
  -> financial-modal-controller integrated open path
  -> clone current visible inputs as modal baseline
  -> render tabs: 월 수입 -> 월 생활비 -> 투자 -> 저축 -> 결과/자동 저축
  -> user edits draft fields / expands one selected child row
  -> derived summary rail updates from draft only
  -> if 저축 + 투자 exceeds available income:
       show same-modal adjustment choices
       block save until choice or resolution
  -> 재무설정 저장
  -> persistence.commitImmediateInputs(nextInputs)
  -> sanitizeInputs() + account repair + totals
  -> renderAll()
  -> summary cards + Sankey update from sanitized durable state

편집 취소 / close
  -> discard baseline/draft state
  -> no commitImmediateInputs()
  -> default summary/Sankey remain unchanged
```

## Common Pitfalls

### Pitfall 1: Treating the SPEC Ordered Flow as Still Authoritative

**What goes wrong:** Planner implements `부부합산`, `본인 설정`, `배우자 설정`, spouse income editing, or household mode controls. [VERIFIED: 10.5-CONTEXT.md]

**Why it happens:** `10.5-SPEC.md` contains the older ordered flow, but `10.5-CONTEXT.md` explicitly overrides it. [VERIFIED: 10.5-CONTEXT.md]

**How to avoid:** Use the adjusted order `월 수입`, `월 생활비`, `투자`, `저축`, `결과/자동 저축`. Add tests that assert couple UI is absent. [VERIFIED: 10.5-CONTEXT.md]

**Warning signs:** Selectors or visible copy for `data-household-overview`, `부부`, `본인 설정`, `배우자 설정`, `spouseMonthlyIncome`, or `household-mode-toggle` appear in Phase 10.5 UI tests. [VERIFIED: codebase grep]

### Pitfall 2: Breaking Draft Save/Cancel

**What goes wrong:** Editing a field mutates durable localStorage before `재무설정 저장`, or close/cancel leaves changed data. [VERIFIED: codebase grep]

**Why it happens:** Current `baselineInputs` is mutable in `financial-modal-controller.js`, and household context updates currently mutate `baselineInputs` before final save while relying on no commit until save. [VERIFIED: codebase grep]

**How to avoid:** Keep a clearly named draft object separate from immutable baseline. On cancel/close, drop draft; on save, construct `nextInputs` and call `commitImmediateInputs()` once. [VERIFIED: codebase grep]

**Warning signs:** Tests need `localStorage.setItem` expectations before save; repeated save creates duplicate items; cancel test fails. [VERIFIED: codebase grep]

### Pitfall 3: Losing Variable-Only `actualSpent`

**What goes wrong:** Fixed expense rows gain actual-spending controls or variable rows lose `actualSpent` after save. [VERIFIED: codebase grep]

**Why it happens:** `sanitizeExpenseItems()` intentionally preserves `actualSpent` only for variable expense items and strips it from fixed items. [VERIFIED: codebase grep]

**How to avoid:** Render target/actual only for `isVariableExpenseItem()` rows under `월 생활비`; write `actualSpent` only onto those draft items. [VERIFIED: codebase grep]

**Warning signs:** Fixed row `rent` retains `actualSpent` after save; tests expecting variable actual preservation fail. [VERIFIED: codebase grep]

### Pitfall 4: Sankey Drift or Duplicate Branches

**What goes wrong:** Modal save creates duplicate items, duplicate Sankey nodes, or branches that do not match sanitized totals. [VERIFIED: 10.5-CONTEXT.md]

**Why it happens:** Creating rows directly or applying adjustment choices multiple times can append instead of replacing draft category arrays. [ASSUMED]

**How to avoid:** Use replace-style updates by category arrays, then let `commitImmediateInputs()` sanitize and `renderAll()` rebuild Sankey from current state. [VERIFIED: codebase grep]

**Warning signs:** Existing tests around `총수입`, `data-financial-category` counts, repeated save, or basic/detail Sankey labels regress. [VERIFIED: codebase grep]

### Pitfall 5: Mobile Horizontal Overflow

**What goes wrong:** The summary rail plus tab content becomes a compressed table at 390px or buttons/text overlap. [VERIFIED: 10.5-CONTEXT.md]

**Why it happens:** Existing CSS already has several grid and overflow overrides; adding another side-by-side rail without mobile stacking will exceed viewport width. [VERIFIED: codebase grep]

**How to avoid:** On mobile, make the summary rail a compact stacked or horizontally wrapping header inside the modal, with one visible selected row editor. Verify `documentElement.scrollWidth - clientWidth <= 4`. [VERIFIED: codebase grep]
