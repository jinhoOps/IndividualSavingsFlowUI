# Phase 10.5: Financial Settings UX Integration Repair - Specification

**Created:** 2026-06-23
**Ambiguity score:** 0.15 (gate: <= 0.20)
**Requirements:** 5 locked

## Goal

Step 1 financial settings changes from split, audit-failing household budget surfaces into one integrated `재무설정 상세` flow that keeps the default screen low-fatigue while exposing dense editing detail only inside the modal.

## Background

Phase 10 shipped the data foundation for household context and adaptive budgeting: `apps/main/modules/household-budget.js` calculates household income, variable expense target/actual/remaining/status, and month-end projection; `apps/main/modules/input-sanitizer.js` preserves the household and actual-spending fields at the Step 1 boundary; `apps/main/modules/financial-modal-controller.js` already owns draft edit/save/cancel behavior for income, expense, savings, and investment categories.

The UI is not yet coherent. The Phase 10 UI review scored 9/24. It found that the planned household budget panel, the implemented financial modal, and the tests contradict each other: the current tests assert that `#householdBudgetPanel` does not exist, while the Phase 10 UI-SPEC expected a separate household budget panel. The repair decision is to not restore a separate household budget panel. Instead, the default Step 1 screen stays summary-first and shows only essential financial state plus a next action, while the detailed edit surface becomes one ordered `재무설정 상세` flow.

## Requirements

1. **Default summary stays light**: The Step 1 default financial settings area must show essential state and one next action without rendering a separate household budget panel or duplicating the full editing surface.
   - Current: `apps/main/index.html` renders `재무 설정 요약` and `#summaryCards`; there is no `#householdBudgetPanel`, and current tests already expect that absence.
   - Target: The default screen keeps a compact summary-first layout and exposes a clear `재무설정 상세` entry action for full editing.
   - Acceptance: A verifier confirms `#householdBudgetPanel` is absent, `#summaryCards` remains before the Sankey section at desktop/tablet/mobile widths, and a visible action opens the financial settings detail flow.

2. **One ordered financial settings detail flow**: The detail modal must present the financial setup in this reading order: `월 수입`, `월 생활비`, `투자`, `저축`, `부부합산`, `본인 설정`, `배우자 설정`.
   - Current: `financial-modal-controller.js` has generic category modals and an outflow mode with `생활비` and `저축-투자` tabs, but the copy and order do not lock the integrated flow.
   - Target: The `재무설정 상세` modal uses the ordered flow above so monthly income, living expenses, investment, savings, household total, and spouse/self settings feel like one connected setup.
   - Acceptance: A verifier opens the detail flow and confirms the visible section/tab order is exactly `월 수입 -> 월 생활비 -> 투자 -> 저축 -> 부부합산 -> 본인 설정 -> 배우자 설정`, with the modal title or primary heading reading `재무설정 상세`.

3. **Variable expenses become living-expense child details**: Variable expense target, actual spending, remaining amount, status, and month-end projection must be visible or editable as child detail rows under `월 생활비`, not as a separate budget panel.
   - Current: `household-budget.js` exposes target/actual/remaining/status/projection calculations, but the UI review found no route exposing `data-household-budget-target`, `data-household-budget-actual`, `월말 예상`, or the projection helper copy to users.
   - Target: `월 생활비` detail rows expose or expand into target, actual, remaining, status, and `월말 예상`; empty living-expense child state is explicit and non-breaking.
   - Acceptance: A verifier can edit a living-expense child row's target and actual values, see remaining amount, status, `월말 예상`, and `현재 사용 속도를 단순 환산한 참고값입니다.`, and sees a clear empty state when no variable living-expense rows exist.

4. **UI audit threshold is high enough to catch fatigue regressions**: The repaired UX must meet a stricter UI review gate: at least 18/24 overall and at least 3/4 each for Copywriting, Visuals, Spacing, and Experience Design.
   - Current: Phase 10 UI review scored 9/24 with Copywriting 1/4, Visuals 1/4, Spacing 2/4, and Experience Design 1/4.
   - Target: Phase 10.5 resolves the blocking UX split and raises density without increasing fatigue, as measured by the follow-up UI review.
   - Acceptance: `$gsd-ui-review` or an equivalent 6-pillar review reports overall score >= 18/24 and Copywriting, Visuals, Spacing, and Experience Design all >= 3/4.

5. **Existing Step 1 data and persistence boundaries stay intact**: The repair must preserve existing sanitizer behavior, modal draft save/cancel behavior, local persistence, and regression-tested Sankey behavior.
   - Current: `financial-modal-controller.js` saves through `persistence.commitImmediateInputs()`, Phase 10 household fields are sanitized at the Step 1 boundary, and `tests/step1.spec.ts` covers targeted Step 1 regressions.
   - Target: The UX repair changes information architecture and controls without creating a new persistence model or bypassing sanitizer/correction flows.
   - Acceptance: Existing Step 1 tests for sanitizer-backed household fields, modal save/cancel, local persistence, and Sankey rendering still pass after the repair; repeated modal save does not create duplicate items, panels, or Sankey branches.

## Boundaries

**In scope:**
- Compact default Step 1 financial summary with a single next action into detail editing.
- One `재무설정 상세` modal or equivalent detail surface that connects income, living expenses, investment, savings, household total, self settings, and spouse settings.
- Living-expense child details for variable target, actual, remaining, status, and month-end projection.
- Copy, spacing, and visual cleanup needed to satisfy the Phase 10.5 UI audit threshold.
- Tests or verification updates that align the current test contract with the integrated UX decision.

**Out of scope:**
- Restoring a separate `#householdBudgetPanel` default-screen panel - this phase explicitly chooses integrated financial settings instead.
- Building Phase 11 pasted bank/card notification capture - that is the next roadmap phase.
- Building Phase 12 dual shared-data merge - this phase can show household setup, but does not merge two Step 1 hashes.
- Building Phase 14 real-estate affordability planning - household income may be reused later, but lending/DSR/LTV flows are not part of this repair.
- Replacing Step 1 persistence, storage, sanitizer, or Sankey architecture - this phase repairs UX on top of the existing model.

## Constraints

- The app remains a static/offline-first browser app; no live account scraping, external finance API, or backend service is introduced.
- Money inputs continue using the existing won-unit parsing, formatting, sanitizer, and local persistence conventions.
- Detail edits continue using draft save/cancel behavior; default-screen summaries must not mutate data by themselves.
- The default screen must not become a dense editor; dense controls belong in the modal or expansion panels.
- Mobile layout must avoid horizontal overflow and must keep text inside controls legible at narrow Step 1 viewports.

## Acceptance Criteria

- [ ] `#householdBudgetPanel` is not rendered on the default Step 1 screen.
- [ ] The default Step 1 financial summary remains before the Sankey section and exposes one clear action into `재무설정 상세`.
- [ ] The detail flow reads in this order: `월 수입`, `월 생활비`, `투자`, `저축`, `부부합산`, `본인 설정`, `배우자 설정`.
- [ ] `월 생활비` child details expose target, actual spending, remaining amount, status, and `월말 예상`.
- [ ] The projection helper copy `현재 사용 속도를 단순 환산한 참고값입니다.` is visible where month-end projection appears.
- [ ] Empty living-expense child state renders a clear non-breaking empty message.
- [ ] Saving the detail modal persists through the existing Step 1 persistence boundary; canceling closes without committing draft changes.
- [ ] Repeated save does not create duplicate items, duplicate household sections, or duplicate Sankey branches.
- [ ] Existing Step 1 sanitizer, persistence, and Sankey regression tests pass.
- [ ] Follow-up UI review score is >= 18/24 overall.
- [ ] Follow-up UI review scores Copywriting, Visuals, Spacing, and Experience Design each >= 3/4.
- [ ] The UI does not use guilt, pressure, or blame copy when showing spending status.
- [ ] The spouse/self setup copy does not imply one partner is responsible for household shortfalls.
- [ ] `자동 저축` is presented as a derived/automatic result, not as a manually editable item in the same way as explicit income, expense, investment, or savings rows.
- [ ] The repaired financial settings surface does not present outputs as legal, lending, tax, or investment advice.
