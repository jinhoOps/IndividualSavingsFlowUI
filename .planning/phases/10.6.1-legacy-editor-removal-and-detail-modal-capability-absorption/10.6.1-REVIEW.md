## Summary

Financial Detail Modal이 legacy editor를 대체하는 경로를 중심으로, ordinary editing, 수입 배분, 저축 만기/수익률, 저장 후 계산 반영 흐름을 검토했습니다. legacy editor 제거 자체는 주요 참조가 제거되어 보이지만, 현재 구현에는 실제 데이터 일관성/계산 오류가 있습니다.

## Critical Issues

### CR-01: 수입 배분 합계가 수입보다 작아도 저장되어 Sankey/계좌 흐름에서 수입이 사라집니다

**File:** `apps/main/modules/financial-modal-controller.js:582`

**Issue:** `validateDraftItem()`은 수입 배분 합계가 수입 금액을 초과하는 경우만 막습니다. 그래서 사용자가 월수입 4,200,000원에 대해 배분을 3,000,000원 + 500,000원처럼 적게 입력하면 저장이 통과합니다. 이후 `buildMonthlySnapshot()`은 수입 총액을 `item.amount`로 유지하지만, 계좌/Sankey 쪽은 `allocations` 합계만 계좌 유입으로 사용하므로 미배분 700,000원이 계좌 흐름에서 조용히 누락됩니다. ordinary editing을 모달로 흡수한 뒤 income allocation이 깨지는 직접 회귀입니다.

**Fix:**
```javascript
const allocated = sumAllocations(allocations);
if (allocated !== amount) {
  setRowError(category, index, "수입 금액과 배분 합계가 일치해야 합니다.");
  return false;
}
```

또는 UX상 미배분을 허용하려면 저장 전에 차액을 기본 `accountId`로 자동 보정해 `allocations` 합계가 항상 `amount`와 같도록 정규화해야 합니다.

### CR-02: 잉여현금 자동 이체가 대상 버킷의 원본 배열 인덱스를 사용해 잘못 배분됩니다

**File:** `apps/main/modules/calculator.js:345`

**Issue:** `targetBuckets`는 필터링된 `targetSavings + targetInvests` 배열인데, 실제 배분 적용 시 `savingsBuckets.indexOf(bucket)`와 `investBuckets.indexOf(bucket)`를 `surplusAdds` 인덱스로 사용합니다. 선택된 계좌에 연결된 저축/투자 항목이 원본 배열의 0번째가 아니면 `surplusAdds[idx]`가 `undefined`가 되어 금액이 빠지거나, 투자 쪽은 `targetSavings.length + idx`가 잘못된 슬롯을 읽습니다. 결과적으로 잉여현금을 특정 계좌로 자동 이체한다는 설정과 시뮬레이션 결과가 달라집니다.

**Fix:**
```javascript
targetSavings.forEach((bucket, localIndex) => {
  bucket.balance += surplusAdds[localIndex] || 0;
});
targetInvests.forEach((bucket, localIndex) => {
  bucket.balance += surplusAdds[targetSavings.length + localIndex] || 0;
});
```
