## User Constraints (from CONTEXT.md)

### Locked Decisions

- **D-01 Primary editor boundary:** Financial Detail Modal is the only primary ordinary Step 1 financial editing surface.
- **D-02 Summary surface boundary:** The Step 1 summary surface remains scan-and-open only; it should not host ordinary financial item editing controls.
- **D-03 Legacy editor removal:** Hidden legacy editor DOM, item-editor controller wiring, and global item-editor pending bar are removal targets unless a narrow compatibility fallback is proven necessary.
- **D-04 Capability absorption:** Useful legacy-only capabilities must move into Financial Detail Modal row edit states before their old surface is removed.
- **D-05 Income allocation location:** Income account allocation belongs inside the income row edit state; compact read rows show a representative deposit account only.
- **D-06 Savings maturity semantics:** Savings maturity month stops future monthly contributions after the maturity month while retaining accumulated balance in projections.
- **D-07 Item-level savings yield location:** Item-level savings yield belongs with maturity month inside collapsed savings row additional settings.
- **D-08 Language cleanup:** User-facing copy must avoid "main editor", "primary editor", "new modal", "advanced editor", "manual transfer", "separate transfer rule", "display-only maturity", and "global savings yield" when those terms conflict with this phase's model.
- **D-09 Regression boundary:** Sanitizer, persistence, Sankey rendering, share/import, and Phase 10.6 row-editing/pending-bar behavior must remain the durable compatibility boundary.

## Summary

Phase 10.6.1 should be planned as a focused removal-and-absorption refactor: remove ordinary Step 1 access to the legacy item editor and global pending controls, then make the Financial Detail Modal cover the remaining capabilities. The ADR explicitly says hidden legacy editor DOM, `item-editor-controller`, the global item-editor pending bar, and tests requiring the secondary editor are not part of the target UX. [VERIFIED: docs/adr/0001-financial-detail-modal-is-the-only-primary-editor.md]

The useful legacy-only capabilities are concentrated in three places: income allocation rows in `list-renderer.js`/`item-editor-controller.js`, savings `annualRate`, and savings/invest `maturityMonth`. [VERIFIED: apps/main/modules/item-editor-controller.js] [VERIFIED: apps/main/modules/list-renderer.js] The modal already has draft state, baseline-vs-draft dirty detection, row edit rendering, modal-local pending bar, and `commitImmediateInputs()` save behavior, so implementation should extend those seams rather than keeping a second editor alive. [VERIFIED: apps/main/modules/financial-modal-controller.js] [VERIFIED: apps/main/modules/persistence-controller.js]

**Primary recommendation:** Make `financial-modal-controller.js` the only Step 1 ordinary editor, migrate income allocation and savings additional settings into modal row edit states, then delete or isolate the legacy editor wiring and update Playwright tests to prove absence of the old path plus preservation of sanitizer/persistence/Sankey/share/import contracts.

### Pattern 1: Modal-Only Draft Mutation

**What:** Keep all ordinary financial item edits in `financial-modal-controller.js` draft state until modal `적용`. [VERIFIED: apps/main/modules/financial-modal-controller.js]  
**When to use:** Any income, expense, savings, invest, or account edit initiated from Step 1 normal UX. [VERIFIED: 10.6.1-CONTEXT.md]  
**Example:**

```javascript
// Source: apps/main/modules/financial-modal-controller.js
function save() {
  const invalid = findFirstInvalidDraftItem();
  if (invalid) {
    editingIndex = invalid;
    renderRows();
    syncFinancialModalPendingBar();
    return;
  }
  const nextInputs = getCandidateDraftInputs();
  persistence.commitImmediateInputs(nextInputs);
}
```

### Pattern 2: Preserve Sanitizer As Durable Boundary

**What:** Modal saves should pass raw drafts to `commitImmediateInputs()`, which sanitizes and repairs the durable state. [VERIFIED: apps/main/modules/persistence-controller.js]  
**When to use:** When adding allocation arrays, `maturityMonth`, or item-level `annualRate` to modal drafts. [VERIFIED: apps/main/modules/input-sanitizer.js]  
**Example:**

```javascript
// Source: apps/main/modules/persistence-controller.js
function commitImmediateInputs(inputs, options = {}) {
  state.inputs = sanitizeInputs(inputs);
  saveState(state.inputs, options);
  setPendingBarVisible(false);
  renderAll();
}
```

### Pattern 3: Modal Row Additional Settings

**What:** Savings-specific `annualRate` and `maturityMonth` belong in savings row edit state, preferably inside a collapsed additional settings area. [VERIFIED: 10.6.1-CONTEXT.md]  
**When to use:** Savings rows only; investment rows should not acquire this phase's savings maturity UI because D-06 specifically defines savings maturity semantics. [VERIFIED: 10.6.1-CONTEXT.md]

### Anti-Patterns to Avoid

- **Keeping legacy editor tests as compatibility proof:** Tests that import `item-editor-controller.js` or rely on `#incomeList` edit selectors preserve the old UX rather than proving its removal. [VERIFIED: tests/step1.spec.ts]
- **Bypassing sanitizer for modal saves:** Directly mutating localStorage would skip `sanitizeInputs()`, share/import normalization, account repair, and monthly total recalculation. [VERIFIED: apps/main/modules/persistence-controller.js]
- **Adding a second pending bar inside or outside the modal:** Phase 10.6 already verified the modal-local pending bar; `#pendingBar` is the legacy global control targeted for removal. [VERIFIED: 10.6-VERIFICATION.md] [VERIFIED: apps/main/index.html]
- **Changing projection semantics in the UI layer:** `maturityMonth` and item-level yield already belong to sanitizer/calculator data flow; UI should edit fields, not duplicate projection rules. [VERIFIED: apps/main/modules/input-sanitizer.js] [VERIFIED: apps/main/modules/calculator.js]

## Common Pitfalls

### Pitfall 1: Removing the Controller Before Absorbing Its Capabilities

**What goes wrong:** Income allocation, savings maturity, or item-level yield become uneditable. [VERIFIED: apps/main/modules/item-editor-controller.js]  
**Why it happens:** Those controls are currently richer in legacy list-renderer/item-editor than in modal row edit state. [VERIFIED: apps/main/modules/list-renderer.js]  
**How to avoid:** Add modal row controls and Playwright coverage first, then delete or isolate legacy wiring.  
**Warning signs:** Tests pass only because `#incomeList` or `#savingsList` still expose edit controls. [VERIFIED: tests/step1.spec.ts]

### Pitfall 2: Breaking Share/Import While Modal Tests Pass

**What goes wrong:** Direct modal save works, but JSON import, share hash, or share-id data loses allocation/yield/maturity fields. [VERIFIED: apps/main/modules/persistence-controller.js]  
**Why it happens:** Tests only inspect localStorage after UI save and skip external normalization paths.  
**How to avoid:** Add `page.evaluate()` tests that build/import/share payloads through `IsfShare`/persistence and assert sanitized fields survive. [VERIFIED: apps/main/modules/persistence-controller.js]

### Pitfall 3: Confusing Legacy Global Pending Bar With Modal Pending Bar

**What goes wrong:** The old `#pendingBar` remains hidden but callable, or tests assert the wrong pending control. [VERIFIED: apps/main/index.html]  
**Why it happens:** Both old global and modal-local pending bars use Korean `취소`/`적용` buttons. [VERIFIED: apps/main/index.html]  
**How to avoid:** Assert `#pendingBar` and `#mobileEditorFab` are absent or unreachable, while `#financialModalPendingBar` still passes Phase 10.6 behavior. [VERIFIED: tests/step1.spec.ts]

### Pitfall 4: Leaving Event-Binding Null References

**What goes wrong:** App boot fails after removing `createItemEditorController` because `bindStep1Events()` still calls `commands.itemEditor.*`. [VERIFIED: apps/main/modules/event-bindings.js]  
**Why it happens:** Bootstrap currently constructs and passes `itemEditor` into event bindings. [VERIFIED: apps/main/modules/bootstrap-controller.js]  
**How to avoid:** Plan bootstrap/event-bindings/UI-controller cleanup in the same wave as controller deletion, or keep a narrow no-op compatibility adapter only if necessary.
