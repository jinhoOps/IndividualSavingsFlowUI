**Mapped:** 2026-06-26

### Account/Allocation Correction Path

- `apps/main/modules/account-correction.js:56-73` recommends fallback account IDs.
- `apps/main/modules/account-correction.js:88-104` repairs item `accountId` values.
- `apps/main/modules/account-correction.js:157-233` rewrites income `allocations` to match amounts.
- `apps/main/modules/account-correction.js:236-255` returns repaired Step 1 inputs plus `accountCorrections`.
- Phase 10.7 should not let this path influence primary Step 1 Sankey or UI. If retained, it should be migration-only or moved to Portfolio/account-flow destination.

```javascript
// apps/main/modules/account-correction.js:236-255
export function repairAccountConnections(inputs) {
  const base = inputs && typeof inputs === "object" ? { ...inputs } : {};
  const corrections = [];
  const accounts = getAccounts(base).map((account) => ({ ...account }));
  const repairContext = { ...base, accounts };

  const repaired = {
    ...base,
    accounts,
    splitIncomeAccounts: Boolean(base.splitIncomeAccounts),
  };

  repaired.incomes = (Array.isArray(base.incomes) ? base.incomes : [])
    .map((income) => repairIncomeItem(repairContext, income, corrections));
  repaired.expenseItems = repairAllocationItems(repairContext, base.expenseItems, "expense", corrections);
  repaired.savingsItems = repairAllocationItems(repairContext, base.savingsItems, "savings", corrections);
  repaired.investItems = repairAllocationItems(repairContext, base.investItems, "invest", corrections);
  repaired.accountCorrections = corrections;

  return repaired;
}
```

  try {
    const snapshot = await IsfStorageHub.getLatestStep1Snapshot();
    if (!snapshot) return null;

    const data = snapshot.data || {};
    let investCapacity = 0;
    if (typeof data.monthlyInvest === 'number') {

## Constraints for Planner

- Do not restore a second Step 1 editor. `docs/adr/0001-financial-detail-modal-is-the-only-primary-editor.md` and `tests/step1.spec.ts:294-376` enforce modal-only editing.
- Step 1 primary mental model must be `income -> expense/savings/invest`. Planner should require `sankey-builder.js` to stop importing `account-correction.js`.
- Legacy `accounts`, `allocations`, `splitIncomeAccounts`, `surplusTransferAccountId`, and `transfers` may be read for migration/handoff, but must not be live Step 1 UI state or primary Sankey state.
- Prefer Portfolio as the first destination unless implementation research finds a blocker: Portfolio already owns stateful allocation-style editing, calculator validation, detail modal editing, and local persistence.
- If using Portfolio, extend `apps/portfolio/modules/state.js`, `calculator.js`, `dom.js`, and `step1-connector.js`; avoid introducing Step 1-owned account-flow state.
- If using a separate page, add every route layer: HTML, entry bridge, app/module/style files, `vite.config.ts` input, `app-header` launcher, and tests.
- Keep storage through `window.IsfStorageHub`/`IsfStorageHub`; do not create new direct localStorage keys from random feature modules except through a state/storage helper matching `apps/portfolio/modules/state.js`.
- Use DOM registries (`apps/main/modules/dom.js`, `apps/portfolio/modules/dom.js`) rather than scattered `document.getElementById` in new controller code.
- Any sanitizer change must preserve imported/shared old data enough to avoid broken page load. Failing legacy account data should degrade to simple Step 1 data, not block rendering.
- DataHub currently shows simulation list only for Step 2 (`shared/components/data-hub-modal.js:199-202`, `275-313`). Account-flow destination list behavior would need explicit modal support if persisted records are user-selectable.

## Risks

- `buildSankeyData` has 11 callers and direct Playwright coverage; changing return shape can break `sankey-renderer.js`, `render-orchestrator.js`, `event-bindings.js`, and `visualization-controller.js`. Preserve `nodes`, `links`, `transfers`, `totalValue`, `splitGroups`, and compatible flags unless planner schedules renderer updates together.
- `sanitizeInputs` is called by state, persistence, event bindings, item editor, visualization, and tests. Removing account fields too aggressively can break imports or old saved data. Prefer a compatibility extraction helper over raw deletion.
- `financial-modal-controller.js` is large and account/allocation logic is interleaved with income editing. Planner should isolate removals around category config, income allocation UI, account selects, and draft field handlers; avoid unrelated modal refactors.
- `account-correction.js` is currently imported by both sanitizer and modal. Moving it requires removing or replacing all call sites, not just hiding UI.
- Portfolio currently persists only portfolio records, not account-flow graphs. If account-flow is added there, define a separate state subtree/key or a clearly versioned record shape before UI work.
- New route option has more blast radius than Portfolio integration: HTML, entry, Vite, header navigation, app code, styles, tests, and DataHub behavior all need updates.
- Existing Step 1 tests still encode Phase 09 account topology. Planner must explicitly schedule test rewrites, not only code changes.
