## Executive Recommendation

Use **Portfolio integration as the Phase 10.7 destination boundary**, not a brand-new page. Step 1 should be simplified now to a primary graph shaped as `수입 -> 소비/저축/투자` with optional detail grouping beneath those categories, while old account/allocation fields are extracted into a `legacyAccountFlow`/handoff-style payload that Step 1 can carry through import/export but does not render, edit, correct, or use for Sankey generation. [VERIFIED: CodeGraph, .planning/codebase/ARCHITECTURE.md]

The safest implementation is to introduce a simple Step 1 Sankey builder path, then remove account controls from Step 1 UI/render orchestration. `buildSankeyData()` currently creates account nodes, resolves invalid account IDs, builds income allocation links, and generates greedy automatic transfers labeled `자동 잔액 맞춤`; changing only copy or CSS would leave the fragile behavior in the primary path. [VERIFIED: CodeGraph node apps/main/modules/sankey-builder.js]

For Phase 10.7, Portfolio should receive only a documented handoff capability, not a complete rich account-flow editor. A full separate app/page would require a new HTML route, new entry bridge, Vite input, app-header navigation, new state/storage, CSS, and fresh E2E tests. Portfolio already has a route, state class, local persistence key, creator/detail modal patterns, and a Step 1 connector, so it is the lower-cost and lower-risk destination for preserved account-flow data. [VERIFIED: .planning/codebase/ARCHITECTURE.md; .planning/codebase/STRUCTURE.md; CodeGraph node apps/portfolio/app.js; CodeGraph node apps/portfolio/modules/state.js; CodeGraph node apps/portfolio/modules/step1-connector.js]

## Boundary Decision: Portfolio vs Separate App

**Recommendation: Portfolio integration.**

Portfolio is the better destination for Phase 10.7 because:

- It is already a separately addressable Vite page with `apps/portfolio/index.html`, `src/entries/step3.ts`, and Rollup input `portfolio`. [VERIFIED: .planning/codebase/ARCHITECTURE.md; rg vite.config.ts]
- It already owns portfolio/allocation-style user workflows and local persistence, while Step 1 owns household cash-flow input and Sankey visualization. [VERIFIED: .planning/codebase/ARCHITECTURE.md; .planning/codebase/STRUCTURE.md]
- It already has `apps/portfolio/modules/step1-connector.js`, although the connector currently imports only investment capacity from the latest Step 1 snapshot. [VERIFIED: CodeGraph]
- It avoids adding a fourth route during a corrective phase whose main purpose is to reduce Step 1 complexity before Phase 11 resumes. [VERIFIED: ROADMAP.md]

A separate app/page should be deferred unless Portfolio cannot host the account-flow domain after a future UX spec. New pages are supported by the architecture, but require `apps/<new-step>/index.html`, `src/entries/<new-step>.ts`, `apps/<new-step>/app.js`, modules, styles, `vite.config.ts`, `shared/components/app-header.js`, and new tests. That cost is unjustified for Phase 10.7 because the immediate need is extraction and boundary documentation, not a mature standalone product. [VERIFIED: .planning/codebase/STRUCTURE.md]

Suggested destination shape for Phase 10.7:

- Add a Portfolio-side "account-flow handoff available" state or connector method that can read preserved legacy account/allocation data from the latest Step 1 data or share envelope. [ASSUMED]
- Keep rich bidirectional account-flow editing out of scope for this phase unless the plan can implement it without touching Step 1 primary rendering again. [VERIFIED: 10.7-CONTEXT.md]
- Add main-to-Portfolio guidance from Step 1 only as a navigation/handoff CTA, not an embedded account editor. [VERIFIED: 10.7-CONTEXT.md]

## Migration and Compatibility Strategy

Use a **sidecar preservation strategy**:

1. On import/load/sanitize, detect legacy account-flow fields:
   - `accounts`
   - `incomes[].accountId`
   - `incomes[].allocations`
   - `expenseItems[].accountId`
   - `savingsItems[].accountId`
   - `investItems[].accountId`
   - `splitIncomeAccounts`
   - `surplusTransferAccountId`
   - `transfers`
   - `accountCorrections`
2. Copy those fields into a legacy handoff container before simplifying the Step 1 primary model. Suggested name: `legacyAccountFlow` or `accountFlowHandoff`. [ASSUMED]
3. Strip or ignore those fields for Step 1 primary rendering, modal editing, and Sankey generation.
4. Preserve non-account financial fields on items: `id`, `name`, `amount`, `group`, `actualSpent`, `varianceAmount`, `annualRate`, `maturityMonth`. These fields support existing Step 1 behavior outside account-flow management. [VERIFIED: CodeGraph node input-sanitizer.js; CodeGraph node calculator.js]
5. Export/share/back up the sidecar only for compatibility and Portfolio handoff. Do not rehydrate it into Step 1 live state except through an explicit handoff command. [VERIFIED: 10.7-CONTEXT.md]

The typed storage model `src/core/types/models.ts` already represents Step 1 as incomes and allocation items without account/allocation fields, which aligns with the simplified canonical model. The legacy JavaScript payload is more account-rich than the typed model. [VERIFIED: CodeGraph node src/core/types/models.ts]

Avoid destructive migration on first load. Old saves should still open, but Step 1 should save back a simplified primary model plus preserved sidecar rather than repaired account fields mixed into active financial items. This prevents the current automatic repair from rewriting user data before the account-flow destination consumes it. [VERIFIED: 10.7-CONTEXT.md; CodeGraph node input-sanitizer.js]

## Automatic Correction Strategy

Automatic account correction should be **removed from Step 1 primary sanitize/render** and retained only if a migration/handoff helper needs it later.

Current correction behavior is too invasive for the new boundary:

- `sanitizeInputs()` always calls `repairAccountConnections()`, so saved, imported, shared, and rendered Step 1 data are repaired before any primary flow uses them. [VERIFIED: CodeGraph node input-sanitizer.js]
- `repairAccountConnections()` creates default accounts, resolves missing account IDs, rebalances income allocations, and emits `accountCorrections`. [VERIFIED: CodeGraph node account-correction.js]
- `sankey-builder.js` imports `recommendAccountId()` and resolves account IDs again while building the graph. [VERIFIED: CodeGraph node sankey-builder.js]
- `visualization-controller.js` exposes a correction refresh path that calls `sanitizeInputs()` and opens account/category modal destinations. [VERIFIED: CodeGraph node visualization-controller.js]

Recommended implementation boundary:

- Remove `repairAccountConnections()` from `sanitizeInputs()` for the Step 1 canonical model.
- Remove `recommendAccountId()` from the primary Sankey builder.
- Delete or quarantine correction UI handlers and tests from Step 1.
- If the Portfolio handoff needs repaired account data, create a separate helper such as `buildAccountFlowHandoff(rawLegacyData)` that can call account correction explicitly without mutating Step 1 state. [ASSUMED]
- Do not run greedy automatic account balancing in the primary Sankey builder. The label `자동 잔액 맞춤` belongs only in a rich account-flow destination, if anywhere. [VERIFIED: CodeGraph node sankey-builder.js]

## Implementation Risks

| Risk | Why It Matters | Mitigation |
|---|---|---|
| Hidden account-flow code remains active | UI removal alone leaves sanitizer repair, Sankey account resolution, network map rendering, and transfer handlers active. | Change data/build boundaries first, then remove UI. Verify with source-level tests for forbidden Step 1 account-flow markers. |
| Breaking savings/investment projection behavior | Savings maturity and item-level savings yield shipped in 10.6.1 and are not account-flow concepts. | Preserve `annualRate` and `maturityMonth`; only strip account/allocation fields. |
| Import/share old data loss | Old data may contain user account names and allocations that are valuable for Portfolio/account-flow handoff. | Preserve sidecar on import/export/backups; test old payload round trip. |
| Tests encode removed behavior | Existing tests assert account labels in Sankey, account correction, allocation editing, and account selects. | Replace with explicit Phase 10.7 tests that assert absence from Step 1 and preservation in handoff. |
| Portfolio scope creep | Full bidirectional account-flow management could expand Step 3 significantly. | Phase 10.7 should document/add handoff boundary only; rich editor can be a future Portfolio phase. |
| Financial-income warning may depend on account IDs | `calculateAccountFinancialIncomes()` maps warnings by account. | Either convert warnings to aggregate global warning, or defer account-specific warning to Portfolio/account-flow destination. |
| Stale CSS/DOM selectors | `apps/main/styles.css` still has transfer/account/allocation selectors. | Remove only unreachable selectors after source DOM and controllers are simplified; avoid broad CSS churn in first slice. |

   - Refactor `buildSankeyData()` or add a simple builder path that uses `snapshot.incomeBreakdown -> total-income -> category targets`.
   - Remove account nodes, allocation links, account transfers, and greedy balancing from the primary Sankey data.
   - Update renderer tooltip/help text for the simple model.

   - Keep `annualRate`, `maturityMonth`, variable actual/range, groups, and financial totals intact.
   - Update import/share/localStorage tests.

3. **Slice 3 - Financial Detail Modal Account-Concept Removal**
   - Remove account category/tab support from normal modal opening.
   - Remove income allocation editor and account selects from income/expense/savings/invest rows.
   - Preserve non-account edit features from 10.6.1.
   - Rewrite modal tests around absence of account-flow concepts.

4. **Slice 4 - Step 1 Render/UI Cleanup**
   - Remove `계좌흐름도` tab, network map render call, transfer list/select render calls, surplus transfer account select/banner, correction refresh path, and stale account-flow tooltip copy.
   - Clean directly unreachable CSS/DOM only after source paths are removed.
