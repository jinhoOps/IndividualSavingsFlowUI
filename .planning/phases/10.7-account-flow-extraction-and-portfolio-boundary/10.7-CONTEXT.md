# Phase 10.7: Account Flow Extraction and Portfolio Boundary - Context

**Gathered:** 2026-06-26
**Status:** Ready for planning

<domain>
## Phase Boundary

This phase removes account-flow modeling from Step 1's primary financial setup and Sankey generation. Step 1 returns to a simpler `수입 -> 지출(소비/저축/투자)` model. Richer account-flow management must move out of Step 1 and be routed either into Portfolio or into a separate app/page, with a clear handoff boundary before Phase 11 continues.

</domain>

<decisions>
## Implementation Decisions

### Step 1 Flow Simplification

- **D-01:** Step 1 should fully remove the user-facing account concept from the primary financial setup. The mental model is `수입 -> 지출(소비/저축/투자)`, not income deposited into accounts and then distributed through account edges.
- **D-02:** Step 1 Sankey should not be driven by account-flow management. Its primary graph should show income flowing into consumption, savings, and investment categories.
- **D-03:** Existing account/allocation behavior is considered a source of instability because it tangles with automatic correction and can negatively affect Sankey generation.

### Account Flow Destination

- **D-04:** Rich 계좌흐름도 behavior should leave Step 1. The final destination may be Portfolio or a separate app/page, depending on which boundary best fits account-flow management after codebase research.
- **D-05:** The user-facing guidance should prefer `main -> portfolio/account-flow destination`: Step 1 remains the simple financial input surface, while richer account-flow editing is reached from the main flow only when the user needs it.
- **D-06:** Bidirectional editing is allowed at the destination level, but Step 1 should not become the ordinary account-flow editor again.

### Migration and Compatibility

- **D-07:** Existing `account` / `allocation` saved data should be preserved or converted only for Portfolio/account-flow handoff. Step 1 should remove it from its own primary storage/rendering model where possible.
- **D-08:** If legacy account/allocation fields are needed for compatibility, they must be treated as migration/input data, not as live Step 1 UI or Sankey state.

### Automatic Correction

- **D-09:** Automatic account correction should no longer influence Step 1's primary Sankey path. Whether the code is removed, reduced to migration-only, or moved to the account-flow destination depends on what research finds about the safest Step 1 boundary.

### the agent's Discretion

- Decide whether 계좌흐름도 belongs inside Portfolio or in a separate app/page after researching the existing Step 3 Portfolio architecture and the cost of adding a new route.
- Decide whether automatic account correction is deleted from Step 1, retained only for migration/import, or moved to the account-flow destination. The constraint is that it must not keep affecting the primary Step 1 Sankey path.
- Decide the minimal compatibility shape for old saved data, as long as Step 1 itself renders and stores the simplified model.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets

- `apps/main/modules/sankey-builder.js`: Current Step 1 Sankey graph builder. This is the primary place where account-flow dependence must be removed or bypassed.
- `apps/main/modules/account-correction.js`: Current automatic account correction and allocation repair logic. This is the main risk area because it can mutate account/allocation assumptions before Sankey rendering.
- `apps/main/modules/input-sanitizer.js`: Step 1 input normalization boundary. Migration of old `accounts` / `allocations` must pass through this layer or a narrow helper near it.
- `apps/portfolio/app.js`, `apps/portfolio/modules/state.js`, `apps/portfolio/modules/dom.js`, `apps/portfolio/modules/calculator.js`: Candidate destination if 계좌흐름도 is integrated into Portfolio.
- `src/core/storage/CompatibilityBridge.ts` and shared storage globals: Cross-step persistence bridge for moving data from Step 1 to Portfolio/account-flow destination.

### Established Patterns

- Each step is a separate Vite HTML route and entry bridge; a separate account-flow app/page requires new HTML, entry, Vite input, navigation, and tests.
- Step 1 state is module-level and browser-local. Avoid introducing a second live account-flow state inside Step 1.
- Step 3 Portfolio already owns allocation-style workflows and local portfolio persistence, making it a plausible host for richer account-flow management.
- Tests should follow existing Playwright coverage in `tests/step1.spec.ts`; a Portfolio destination may require or justify `tests/step3.spec.ts`.

### Integration Points

- Step 1 summary/detail modal and Sankey rendering should be simplified together, not only hidden visually.
- Portfolio or a new account-flow route should receive data through an explicit handoff/import path, not by Step 1 continuing to expose account editing.
- DataHub/share/import compatibility must be checked so old saved data does not break simplified Step 1.

</code_context>

<specifics>
## Specific Ideas

- User specifically identified the problem as: account functionality was added too early, became tangled with automatic correction, and harmed Sankey generation.
- User prefers removing the account concept from Step 1 entirely and making Step 1 a simpler income-to-consumption/savings/investment flow.
- User is open to richer account-flow management living in Portfolio or a separate app, depending on what fits the account-flow management boundary.
- User wants bidirectional editing allowed in the destination, while the UX nudges users from main toward Portfolio/account-flow destination instead of keeping Step 1 as the editor.
- Existing saved account/allocation data should use option C: preserve/convert only when handing off to Portfolio/account-flow destination; remove it from Step 1's primary model.

</specifics>
