# Phase 10.6: Financial Detail Modal Editing UX Repair - Discussion Log

> **Audit trail only.** Do not use as input to planning, research, or execution agents.
> Decisions are captured in CONTEXT.md — this log preserves the alternatives considered.

**Date:** 2026-06-25
**Phase:** 10.6-financial-detail-modal-editing-ux-repair
**Areas discussed:** Read mode density, row editing behavior, new item flow, pending bar, deletion, modal close, mobile layout, edit fields, variable expense simplification, validation, save path

---

## Read Mode Density

| Option | Description | Selected |
|--------|-------------|----------|
| Name + amount only | Compact row with only item name and amount. Hide duplicate group/category labels. | ✓ |
| Name + amount + one helper label | Show a single helper such as account or group. | |
| Fixed two-line row | Always use a second line for account/group/status. | |

**User's choice:** Name + amount only.
**Notes:** Item names usually carry the account/group meaning already. Repeated labels like `수입, 수입` and `변동비, 변동비` should be removed.

---

## Edit Affordance

| Option | Description | Selected |
|--------|-------------|----------|
| Whole-row click | No required icon; click row to edit. | ✓ |
| Text edit button | A small `편집` button on the right. | |
| Icon button | Keep right-side icon button. | partial |

**User's choice:** Whole-row click, with desktop icon acceptable if it does not cause wrapping.
**Notes:** On mobile, protect compact row density. On desktop, a right-side icon can stay because it does not cause wrapping there.

---

## Row And Tab Switching

| Option | Description | Selected |
|--------|-------------|----------|
| Fold current, keep draft, open new row | Allows continuous multi-row edits before applying. | ✓ |
| Fold current only | Keep draft but do not open the newly clicked row. | |
| Block switching until apply/cancel | Safer but more frustrating. | |

**User's choice:** Fold current, keep draft, open new row. Tab switching also allowed with draft retained.
**Notes:** `적용` saves all accumulated changes; `취소` reverts all accumulated changes.

---

## New Item Flow

| Option | Description | Selected |
|--------|-------------|----------|
| Temporary row at top | Add row near the header button. | |
| Temporary row at bottom | Add draft row at the bottom of the current list. | ✓ |
| Header mini form | Open an add-only mini form below the header. | |

**User's choice:** Temporary row at bottom.
**Notes:** Header-right add button is still preferred, but the new draft row should appear at the list bottom. Empty draft row creation does not show pending bar.

---

## Pending Bar

| Option | Description | Selected |
|--------|-------------|----------|
| Fade/slide down | Short exit animation after cancel/apply. | ✓ |
| Success feedback then disappear | Show a brief status before disappearing. | partial |
| Immediate removal | Remove with no animation. | |

**User's choice:** Fade/slide down, with point-color feedback on apply.
**Notes:** Pending bar appears only after actual draft changes. `취소` reverts draft and hides the bar without closing the modal. `적용` persists and keeps the modal open.

---

## Deletion

| Option | Description | Selected |
|--------|-------------|----------|
| Draft delete | Delete only in draft; apply persists, cancel restores. | ✓ |
| Immediate delete | Permanently delete at click time. | |
| Confirm before draft delete | Ask before marking row deleted. | |

**User's choice:** Draft delete.
**Notes:** Empty temporary rows can disappear immediately without pending state.

---

## Modal Close

| Option | Description | Selected |
|--------|-------------|----------|
| Confirm if pending | Ask before discarding pending changes. | ✓ |
| Always close and discard | Fast but risky. | |
| Block close while pending | Safe but heavy. | |

**User's choice:** Confirm if pending.
**Notes:** Confirmation copy: `변경된 내용을 저장하지 않고 닫으시겠습니까?`

---

## Mobile Layout

| Option | Description | Selected |
|--------|-------------|----------|
| Bottom sticky pending bar | Keep actions near thumbs; add bottom padding. | ✓ |
| Top sticky pending bar | More visible but can conflict with tabs/rail. | |
| Per-row pending actions | Local context but poor for multi-tab accumulated draft. | |

**User's choice:** Bottom sticky pending bar.
**Notes:** Mobile rail should show `수입`, `생활비`, `투자`, `저축`, where `저축` includes explicit savings plus automatic savings. Mobile row names truncate; amount stays visible.

---

## Edit Fields

| Option | Description | Selected |
|--------|-------------|----------|
| Name -> amount -> account/group | Conventional order and clear for new/edit rows. | ✓ |
| Amount -> name -> account/group | Faster for amount-only edits. | |
| Amount first, advanced fields collapsed | Minimal but hides common edits. | |

**User's choice:** Name -> amount -> account/group.
**Notes:** Amount field includes direct input, steppers, and quick increase buttons.

---

## Variable Expense Simplification

| Option | Description | Selected |
|--------|-------------|----------|
| Direct input + quick variance buttons | Edit average amount plus `±범위`; quick buttons `±1만`, `±5만`, `±10만`. | ✓ |
| Quick variance buttons only | Simpler but less flexible. | |
| Percentage variance | Abstract and less natural for household budgets. | |

**User's choice:** Direct input + quick variance buttons.
**Notes:** Compact rows show average value only. Variance appears in edit mode and modal summary, not every row.

---

## Validation

| Option | Description | Selected |
|--------|-------------|----------|
| Row-local errors | Show validation inside the affected editing row. | ✓ |
| Pending-bar errors | Show common errors near apply/cancel. | |
| Both | Row errors plus common summary. | |

**User's choice:** Row-local errors.
**Notes:** Direct input allows 1,000 KRW increments. Buttons remain 10,000 KRW and larger increments.

---

## Save Path

| Option | Description | Selected |
|--------|-------------|----------|
| Pending bar only | Pending bar is the only apply/cancel path when changes exist. | ✓ |
| Keep existing save plus pending bar | Risks duplicated save affordances. | |
| Absorb existing save area into pending bar | Possible but must keep state distinction clear. | |

**User's choice:** Pending bar only.
**Notes:** Do not show a separate active `재무설정 저장` button alongside the pending bar.
