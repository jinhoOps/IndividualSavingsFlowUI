## User Constraints (from CONTEXT.md)

### Locked Decisions

#### Scope And Carry-Forward
- **D-01:** Use Phase 10.6 as a new follow-up phase rather than reopening Phase 10.5.
- **D-02:** Keep the Phase 10.5 integrated `재무설정 상세` direction. This phase polishes the modal editing UX; it does not reintroduce separate household/couple UI.
- **D-03:** Keep the default Step 1 screen light. Dense editing remains inside the modal.

#### Compact Read Mode
- **D-04:** The modal opens in compact read mode. Opening the modal must not show the pending bar.
- **D-05:** Read-mode rows show only `항목명 + 금액` by default.
- **D-06:** Hide repeated category/group labels when they duplicate the row meaning. Avoid outputs like `수입, 수입` or `변동비, 변동비`.
- **D-07:** Account/group/category details are available in edit mode or supporting detail, not required in compact row read mode.
- **D-08:** Remove repeated panel titles such as `financial-detail-panel__title` when the active tab name already communicates the section.
- **D-09:** Remove or hide user-facing traces of implementation language such as "신설 모달".
- **D-10:** Desktop/wide screens may keep a right-side edit icon if it does not cause wrapping. Rows themselves should also be clickable for edit entry.
- **D-11:** Mobile/narrow screens use whole-row click for edit entry. Edit icons may be hidden or made non-layout-affecting to protect `이름 + 금액` density.

#### Row-Level Editing
- **D-12:** Editing is row-level. One selected row expands; other rows remain in compact read mode.
- **D-13:** Expanded edit field order is: 이름 -> 금액 -> 계좌/그룹.
- **D-14:** Existing row values must be prefilled when editing begins. Editing must feel like modifying an existing row, not recreating it.
- **D-15:** Clicking outside the edited row folds it back to read mode.
- **D-16:** If no value changed, folding the row leaves no pending state.
- **D-17:** If values changed, folding the row preserves draft changes and shows the pending bar.
- **D-18:** Clicking another row while changes are pending is allowed: the current row folds, the new row opens, and accumulated draft changes remain pending.
- **D-19:** Tab navigation while changes are pending is allowed. The current row folds, draft changes persist, and the pending bar remains visible.
- **D-20:** `적용` persists all accumulated draft changes across tabs at once. `취소` reverts all accumulated draft changes across tabs.

#### Money Editing
- **D-21:** Money fields support direct input, `+` / `-` steppers, and quick increase buttons.
- **D-22:** `+` / `-` steppers change by 10,000 KRW.
- **D-23:** Quick buttons are increase-only: `+5만`, `+10만`, `+100만`.
- **D-24:** Direct input may use 1,000 KRW increments and must reject invalid non-1,000-unit amounts.
- **D-25:** Money values must not go below 0.
- **D-26:** Direct-input validation copy should be short and local to the row, for example `금액은 1,000원 단위로 입력해 주세요.`

#### Variable Expense Simplification
- **D-27:** Variable expenses should be simplified. Users are expected to enter an average expected amount, not manage heavy actual-spending/projection detail in the primary editing flow.
- **D-28:** Treat variable expenses similarly to fixed expenses in row behavior, with the name/type carrying the meaning.
- **D-29:** Add a separate variance/range field for variable expenses.
- **D-30:** Variance input supports direct entry plus quick buttons `±1만`, `±5만`, `±10만`.
- **D-31:** Individual read-mode rows still show only the average amount. Do not add per-row `±범위` text to compact rows.
- **D-32:** Individual edit mode exposes average amount and `±범위`.
- **D-33:** Show range effects only in the `재무설정 상세` summary area, for example as a living-expense expected range. Do not force range display into every row.

#### Pending Bar
- **D-34:** The pending bar appears only after real draft changes. Edit entry, modal open, tab switch, row selection, and opening an empty new-item row do not count as changes by themselves.
- **D-35:** The pending bar is the only visible path for applying or canceling draft changes. Do not show a separate active `재무설정 저장` button at the same time.
- **D-36:** Pending bar button copy is `취소` / `적용`.
- **D-37:** `취소` reverts draft changes, removes draft-only new rows, folds edit rows, keeps the modal open, and hides the pending bar with a short fade/slide down animation.
- **D-38:** `적용` persists draft changes through the existing save boundary, keeps the modal open, leaves the user on the current tab, folds all rows to read mode, gives a short point-color feedback, then hides the pending bar with fade/slide down.
- **D-39:** On mobile, the pending bar is sticky at the modal bottom. Add enough modal-body bottom padding so it never covers focused fields or the last row.
- **D-40:** Pending bar touch targets must be at least 44px on mobile.

#### New Item Flow
- **D-41:** Each tab's item header owns the add action on the right side, using tab-specific copy such as `수입 추가`, `생활비 추가`, `투자 추가`, or `저축 추가`.
- **D-42:** On mobile, the add action may wrap below the header when needed.
- **D-43:** Clicking add creates a draft-only temporary row at the bottom of that tab's list and opens it in edit mode.
- **D-44:** Creating an empty temporary row does not show the pending bar.
- **D-45:** If the temporary row is still empty and the user clicks away, remove it without showing the pending bar.
- **D-46:** Once meaningful values are entered, the temporary row becomes part of draft changes and the pending bar appears.
- **D-47:** `취소` removes value-filled temporary rows along with other draft changes. `적용` persists them as real rows.

#### Deletion
- **D-48:** Delete is draft-only. Deleting an existing row marks it removed in draft state and shows the pending bar.
- **D-49:** Deleted rows are permanently removed only after `적용`.
- **D-50:** `취소` restores deleted rows.
- **D-51:** Deleting an empty temporary row simply removes it without requiring pending state.

#### Modal Close
- **D-52:** If there are pending changes and the user attempts to close via X, outside click, or ESC, ask: `변경된 내용을 저장하지 않고 닫으시겠습니까?`
- **D-53:** Confirming the close prompt discards draft changes and closes the modal.
- **D-54:** Canceling the close prompt keeps the modal open and preserves the pending bar.
- **D-55:** If there are no pending changes, modal close behaves normally.

#### Mobile Layout
- **D-56:** Mobile summary rail is reduced to a compact 2-line summary.
- **D-57:** Mobile rail fields are `수입`, `생활비`, `투자`, `저축`.
- **D-58:** Mobile `저축` means explicit savings plus automatic savings. Do not split `자동 저축` into a separate mobile rail cell.
- **D-59:** Mobile rows prioritize amount visibility. Use `이름 + 금액`; names truncate to one line if needed; amounts must not wrap, clip, or ellipsize.
- **D-60:** Row height should stay compact and consistent on mobile.

#### Validation And Error Display
- **D-61:** Validation errors appear inside the relevant editing row, not in the pending bar.
- **D-62:** If `적용` finds an error, keep the modal and pending bar open, expand the first invalid row, and show the row-local error.
- **D-63:** Multiple invalid rows can be revealed progressively, but the first error should be actionable immediately.
- **D-64:** Use short row-local copy: `이름을 입력해 주세요.`, `금액은 1,000원 단위로 입력해 주세요.`, `금액은 0원 이상이어야 합니다.`

### Pattern 1: Baseline-Vs-Draft Pending Detection

**What:** create a helper that builds the full candidate draft input object and compares it to `baselineInputs`; do not use edit selection as a proxy for changes. [VERIFIED: current `syncFinancialModalPendingBar()` uses selection/outflow-mode shortcuts]

**When to use:** every `input`, `change`, add/delete, tab navigation, row fold, and apply/cancel action. [VERIFIED: current delegated handlers are centralized in `bind()`]

**Example:**

```javascript
function getDraftInputs() {
  if (isOutflowMode()) {
    return INTEGRATED_CATEGORIES.reduce(
      (inputs, category) => setItemsForCategory(inputs, category, getDraftItemsForCategory(category)),
      baselineInputs,
    );
  }
  return setItemsForCategory(baselineInputs, activeCategory, draftItems);
}

function hasRealDraftChanges() {
  return Boolean(baselineInputs) && JSON.stringify(getDraftInputs()) !== JSON.stringify(baselineInputs);
}
```

### Pattern 2: Apply/Cancel In Place

**What:** `취소` should restore draft state from `baselineInputs`, fold rows, and keep the modal open; `적용` should persist through `commitImmediateInputs()`, reset `baselineInputs` to the newly applied visible data, fold rows, and keep the same tab active. [VERIFIED: Phase 10.6 CONTEXT + current `save()`/`close()` behavior]

**When to use:** replace `dom.financialModalCancel.addEventListener("click", close)` and remove `close()` from the successful `save()` path. [VERIFIED: financial-modal-controller.js]

### Pattern 3: Row-Local Validation

**What:** store validation errors by `{category,index,field}` or equivalent, render the message inside `renderEditingRow()` or variable edit detail, and expand the first invalid row on apply. [VERIFIED: Phase 10.6 CONTEXT requires row-local errors]

**When to use:** direct money input, new temporary rows, delete/apply, and non-1,000 KRW validation. [VERIFIED: current `validateItems()` returns global alert strings]

### Anti-Patterns to Avoid

- **Pending bar on open or selection:** current `isOutflowMode() return true` and `hasEditingSelection()` behavior violates D-04 and D-34. [VERIFIED: financial-modal-controller.js]
- **Closing modal on pending actions:** current cancel binding and successful save path close the modal, violating D-37 and D-38. [VERIFIED: financial-modal-controller.js]
- **Detached create wizard:** current `startCreateFlow()` replaces rows with a multi-step create screen, violating D-41 through D-47. [VERIFIED: financial-modal-controller.js]
- **`window.alert` validation:** current save/create validation uses alerts, violating D-61 through D-64. [VERIFIED: financial-modal-controller.js]
- **Foregrounding actual/projection variable UI:** current living tab shows target, actual, remaining, status, and projection for variable rows; Phase 10.6 supersedes visible emphasis with average plus variance/range while preserving data compatibility. ## Common Pitfalls

### Pitfall 1: Treating Edit Selection As Dirty State

**What goes wrong:** pending bar appears on modal open, row click, or tab switch. [VERIFIED: current `syncFinancialModalPendingBar()`]
**Why it happens:** current code checks `hasEditingSelection()` and hard-codes integrated outflow mode as changed. [VERIFIED: financial-modal-controller.js]
**How to avoid:** compare a full draft input snapshot against `baselineInputs`. [VERIFIED: codebase architecture]
**Warning signs:** tests can open `재무설정 상세` and immediately find `#financialModalPendingBar` visible. [VERIFIED: Phase 10.6 decisions]

### Pitfall 2: Applying Changes By Closing The Modal

**What goes wrong:** user loses tab context after `적용` or `취소`, contrary to 10.6. [VERIFIED: 10.6-CONTEXT.md]
**Why it happens:** current cancel is bound to `close()` and `save()` calls `close()` after commit. [VERIFIED: financial-modal-controller.js]
**How to avoid:** split modal-close, pending-cancel, and pending-apply into separate functions. [VERIFIED: controller binding layout]
**Warning signs:** Playwright expects `#financialModal` to remain visible after pending-bar apply/cancel and fails. [VERIFIED: test strategy]

### Pitfall 3: Breaking Phase 10.5 No-Couple-UI Contract

**What goes wrong:** deferred household controls or labels return while adding variable/range behavior. [VERIFIED: 10.5-CONTEXT.md + 10.6-CONTEXT.md]
**Why it happens:** `household-budget.js` still contains household/spouse data helpers, but they are not a visible-flow license. [VERIFIED: household-budget.js]
**How to avoid:** keep helper compatibility but assert absence of `#householdBudgetPanel`, `[data-household-overview]`, spouse field selectors, and exact labels. [VERIFIED: tests/step1.spec.ts]
**Warning signs:** visible strings `부부합산`, `본인 설정`, or `배우자 설정` appear in the modal. [VERIFIED: 10.5-VERIFICATION.md]

### Pitfall 4: Adding Variance Without Sanitizer Compatibility

**What goes wrong:** a new range field is displayed but lost on save/import, or corrupts old records. [VERIFIED: input-sanitizer.js currently has no variance field]
**Why it happens:** `sanitizeAllocationItems()` only preserves known fields such as `group`, `annualRate`, and optional `maturityMonth`. [VERIFIED: input-sanitizer.js]
**How to avoid:** if implementing D-29/D-30, add a narrowly named non-negative money field for variable expenses only, sanitize it in `sanitizeExpenseItems()`, and strip it from fixed rows. [VERIFIED: sanitizer pattern for `actualSpent`]
**Warning signs:** saved variable expense has the UI range value before save but localStorage loses it after `commitImmediateInputs()`. [VERIFIED: persistence boundary]
