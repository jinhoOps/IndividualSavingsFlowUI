# Phase 10.6 UI Review

## Verdict

PASS with follow-up notes. Phase 10.6 now matches the main intended UX: `재무설정 상세` opens as a compact read-first modal, row editing is local and draft-preserving, money controls no longer force unrelated layout wrapping, the add action lives in the active panel header, group header duplication is removed, and summary expense rows aggregate fixed/variable totals.

The remaining issues are not phase blockers, but they are visible polish/accessibility gaps that should be folded into the next cleanup pass before treating the modal as final.

## Findings

### UXR-10.6-UI-01: Pending-Bar Exit Animation Is Specified But Effectively Disabled

**Severity:** Medium  
**Pillar:** Experience Design / Spacing  
**Evidence:** `syncFinancialModalPendingBar()` assigns one of three animation classes in `apps/main/modules/financial-modal-controller.js:303`, but the financial-modal-specific CSS sets all three classes to `animation: none` in `apps/main/styles.css:1063`.

This means the user-visible requirement "펜딩바만 닫혀야 하고 사라지는 애니메이션" is behaviorally correct for state, but visually under-delivered. The bar disappears abruptly after cancel/apply.

**Recommended fix:** Replace random entry classes with deterministic states, for example `is-entering`, `is-exiting`, and `is-applied`. Keep cancel/apply modal-local, but let the bar fade/slide down before `hidden=true`. Add one Playwright assertion for class/state transition rather than timing-heavy animation verification.

### UXR-10.6-UI-02: Add Menu Lacks Menu Semantics And Keyboard Close

**Severity:** Medium  
**Pillar:** Visuals / Experience Design  
**Evidence:** The `+` trigger has `aria-label` and `aria-expanded`, but no `aria-haspopup`, menu role, item role, or Escape handling in `apps/main/modules/financial-modal-controller.js:991`.

For mouse users this is likely fine, but keyboard and assistive-tech users get a button that opens more buttons without the expected menu contract. Since this control now owns item/group creation, it should be more robust.

**Recommended fix:** Add `aria-haspopup="menu"` to the trigger, `role="menu"` to `.financial-add-menu__panel`, and `role="menuitem"` to menu buttons. Close on Escape and on focus leaving the menu. Keep the current compact visual design.

### UXR-10.6-UI-03: Account Read Rows Still Show Low-Value Repeated Metadata

**Severity:** Low  
**Pillar:** Copywriting / Spacing  
**Evidence:** Account rows render the account name as the strong title, then add a repeated `계좌 별칭` summary label in `apps/main/modules/financial-modal-controller.js:704`.

This does not violate the latest explicit request to remove the account alias edit icon, and the icon is removed for account rows. Still, as a compact read-mode row, repeating `계좌 별칭` under every account name has low information value.

**Recommended fix:** In account read mode, replace the repeated label with useful account metadata if available, or omit the summary line entirely. Keep the edit-mode field label as `계좌 별칭`.

### UXR-10.6-UI-04: Quick Amount Buttons Avoid Wrapping But May Hide Affordance On Very Narrow Widths

**Severity:** Low  
**Pillar:** Spacing  
**Evidence:** The quick row uses `flex-wrap: nowrap` with horizontal overflow in `apps/main/styles.css:1028`.

This correctly prevents the original line-wrapping bug and isolates quick buttons from other fields. The tradeoff is that on very narrow screens the third quick action may require horizontal scroll without a strong scroll affordance.

**Recommended fix:** Keep the independent quick row, but consider either a two-row fixed mini-grid at mobile widths or a subtle clipped-edge/scroll affordance. Do not let quick buttons reflow into the main input row.

## Non-Blockers / Accepted Tradeoffs

- The hidden Sankey correction control still exists in the DOM because account correction remains part of the data/repair pipeline, but it is not visible in the chart header and is covered by tests.
- Variable expense range appears in summary/edit contexts rather than compact rows, which matches the Phase 10.6 simplification decision.
- Mobile uses whole-row click and hides the edit icon from layout pressure; this is the right tradeoff for amount visibility.

## Recommended Next Pass

1. Implement deterministic pending-bar exit/apply feedback and test its state classes.
2. Harden the `+` add menu with menu ARIA and Escape/focus-close behavior.
3. Remove or improve the repeated account-row `계좌 별칭` read-mode summary.
4. Add a narrow-width visual regression around quick amount overflow affordance if future UI tooling supports screenshots.

## Final Assessment

Phase 10.6 should be considered UI-pass for the core user complaints: pending state no longer appears prematurely, modal-local apply/cancel behavior is correct, money controls are no longer polluting adjacent layout, duplicate group/header text is removed, and the financial summary now presents aggregate expense meaning. The remaining items are polish/accessibility follow-ups, not blockers to closing the phase.
