## Phase 10 Screen Contract

### Default Step 1 Screen

The default Step 1 screen must remain summary-first:

1. Existing Step 1 financial summary groups stay above the Sankey.
2. The Step 1.2 `신혼부부 예산` panel appears in the summary area before the Sankey, not below controls.
3. Sankey stays directly below the summary area.
4. Controls and projection remain after Sankey.
5. Detailed category rows and item editing must not appear on the default screen.

The `신혼부부 예산` panel contains exactly:

| Element | Contract |
|---------|----------|
| Title | `신혼부부 예산` |
| Status badge | One short badge: `여유`, `주의`, or `초과` |
| Metric 1 | `가구 월수입` with household monthly income |
| Metric 2 | `변동비 실제/목표` as actual / target |
| Metric 3 | `남은 변동비` as target - actual, allowing negative display for overspend |
| CTA | `예산 상세 편집` opens modal |

Do not add a fourth metric, default-screen item table, category drilldown table, chart, pasted-text capture, merge controls, or real-estate calculator in Phase 10.

### Summary Panel Layout

| Viewport | Layout |
|----------|--------|
| Desktop >= 769px | Flat panel with title/badge/CTA row and a 3-column metric grid |
| Mobile <= 768px | Title/badge/CTA wrap safely; metrics stack as compact rows or a single-column grid |

Dimensions:
- Summary panel padding: 16px desktop, 16px mobile.
- Metric card min-height: 96px desktop, auto but at least 72px mobile.
- Metric grid gap: 8px.
- Badge height: compact text badge, not a large pill.
- CTA min-height: 40px desktop, 44px mobile.

### Detailed Modal

The modal is the only working area for Step 1.2 details.

Required sections:

| Section | Controls |
|---------|----------|
| Household context | Segmented control for `1인 소득` and `맞벌이`; spouse monthly income input shown/enabled only when relevant |
| Budget summary | Total variable target, total actual, remaining, `월말 예상`, and status badge |
| Variable expense rows | Compact cards for variable expenses only; each row shows name, target, actual, remaining/progress, projection |
| Actions | `취소` closes without durable mutation; `저장` commits through the existing persistence boundary |

Modal behavior:
- Open from `예산 상세 편집` or the Step 1.2 panel.
- Use draft state while editing.
- Closing with `취소` or close button discards unsaved household budget edits.
- Only `저장` mutates durable state through `persistence.commitImmediateInputs()`.
- Existing Step 1 financial modal explicit save/cancel behavior must remain unchanged.
- Use DOM APIs and `textContent` for user-controlled names and values.

### Variable Expense Rows

Rows must be compact cards, not a wide table.

Each row contains:
- Expense name.
- Target monthly budget, labeled `목표`.
- This month's actual spending, labeled `실제`.
- Remaining amount, labeled `남은 금액`.
- Progress indicator rendered as a simple horizontal bar or compact percentage text.
- Row status using the same `여유` / `주의` / `초과` rules.

Input constraints:
- Target budget uses existing `expenseItems[].amount` and remains stored in won.
- Actual spending uses the Phase 10 actual field for variable expenses only.
- Actual spending accepts `0` and positive won values.
- Amount displays must use existing `IsfUtils` Korean won formatting helpers.
- Fixed expense rows must not show actual-spending controls.

## Interaction Contract

| Interaction | Required Behavior |
|-------------|-------------------|
| Open panel | Opens `신혼부부 예산 상세` modal without navigating away or changing Sankey |
| Toggle income mode | `1인 소득` works with spouse income absent/0; `맞벌이` allows spouse income input |
| Edit actual spending | Updates modal draft and derived row metrics immediately inside modal only |
| Cancel modal | Restores the default screen with no durable mutation |
| Save modal | Sanitizes, persists, refreshes summary panel, and rerenders Sankey without corrupting existing Step 1 data |
| Overspend | Shows `초과`, negative remaining amount, and non-destructive warning styling |
| No variable expenses | Shows the empty state; CTA remains available |

Keyboard and accessibility:
- The summary panel CTA must be keyboard focusable.
- Modal focus starts on the modal title or first meaningful control.
- All icon-only controls need an accessible label; prefer visible text for Phase 10 actions.
- Badge text must not be color-only; the visible Korean label is required.
- Focus ring uses `rgba(234, 91, 42, 0.34)` or existing primary focus styling.

## Responsive And Density Contract

Phase 10 must preserve compact mobile density:

- No horizontal overflow at 390px viewport width.
- No over-wide default-screen table.
- Modal rows may stack fields on mobile, but each row remains a compact card.
- Buttons and close controls meet 44px touch target on mobile.
- Text inside badges, metric cards, and buttons must not clip or overlap.
- Use `overflow-wrap: anywhere` only for long numeric/label values; do not shrink body text with viewport units.
- The Sankey remains directly below summary content and must not be pushed below large default-screen editors.

---

## Out Of Scope

Do not include:
- Pasted Korean bank/card capture input.
- Dual-ISF household merge UI.
- Historical spending comparison chart.
- Real-estate affordability planner.
- Live banking, scraping, or external financial integrations.
- Separate account-management surface.
- Destructive household reset/delete controls.
