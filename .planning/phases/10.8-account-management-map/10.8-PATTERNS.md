## State and Storage Patterns

### Page-Owned Draft State

Use a separate storage key for Account Map, similar to Portfolio state:

- `apps/portfolio/modules/state.js` lines 3-6 define a page storage key and class.
- Lines 18-29 load stored data with a default fallback.
- Lines 43-46 update `lastUpdated` and call `IsfStorageHub.saveLocal`.

Pattern to copy:

```js
const STORAGE_KEY_ACCOUNT_MAP = "isf-account-map-v1";

export class AccountMapState {
  constructor() {
    this.data = {
      schemaVersion: 1,
      source: null,
      accounts: [],
      relationships: [],
      candidates: [],
      selectedId: "",
      lastUpdated: new Date().toISOString(),
    };
  }
}
```

Keep Account Map saves page-owned in Phase 10.8. Do not write accepted relationships back into Main unless a later plan explicitly defines that contract.

### Main Data Connector

Use the Step 2 connector as the current bridge-aware pattern:

- `apps/simulation/modules/step1-connector.js` lines 9-23: read `isf-rebuild-v1` through `getHubStorage()?.loadLocal`, then fallback to `localStorage`.
- Lines 83-135: compare/import latest Step 1 data without overwriting user-controlled draft values blindly.
- Lines 138-172: resolve latest snapshot through `hub.getLatestStep1Snapshot()` with local fallback.
- Lines 175-187: import latest source, render, mark dirty, and show feedback.

Avoid `apps/portfolio/modules/step1-connector.js` as the primary pattern because it imports old `shared/storage/hub-storage.js` directly at line 5. It is useful only as a small stale reference.

### Sanitized Step 1 Input Contract

Draft builder should consume sanitized fields, not raw arbitrary data:

- `apps/main/modules/input-sanitizer.js` lines 55-129: `sanitizeInputs` normalizes accounts, transfers, income, savings, invest, expense, and repairs account links.
- Lines 144-147: `isVariableExpenseItem` identifies variable expenses; Account Map should exclude these by default.
- Lines 149-200 and 215-242: income items carry `accountId` plus `allocations[]`.
- Lines 546-564: `sanitizeTransfers` keeps only valid account-to-account transfers with `id`, `sourceAccountId`, `targetAccountId`, `amount`, and `label`.

Recommended builder modules:

- `buildAccountMapDraftFromMain(inputs)`
- `buildRelationshipsFromIncome(inputs)`
- `buildRelationshipsFromTransfers(inputs)`
- `buildSavingsInvestmentRelationships(inputs)`
- `buildFixedExpenseCandidates(inputs)`
- `classifyRecurringPaymentCandidate(item)`

Draft relationships should include `sourceRef` and `confidence` so inferred fixed-expense candidates are not presented as bank truth.

## Boundaries and Anti-Patterns

- Do not attach Account Map to Portfolio. Portfolio is not the destination for this phase.
- Do not turn Account Map into a budget/category screen. Ordinary variable expenses and one-off spending stay out of accepted relationships by default.
- Do not create another primary Step 1 income/expense/savings/invest editor. ADR 0001 keeps ordinary financial editing in the Financial Detail Modal.
- Do not remove restored Main account-flow behavior while adding the new route. Main should receive a lightweight entry/mini-map only.
- Do not treat inferred `accountId`, surplus routing, or fixed-expense candidates as real bank/card data. Store `confidence` and `sourceRef`.
- Do not show exact monthly amounts throughout the map overview. Reveal amounts in detail UI after selection.
- Do not use old shared storage modules directly for new Account Map code when the entry installs `src/core/storage/CompatibilityBridge.ts`.
- Do not add relationship filter tabs in this phase. Store `type` metadata now; defer filters.
