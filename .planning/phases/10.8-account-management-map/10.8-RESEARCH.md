## Recommended Account Map Model

Use a page-owned draft model derived from Main, not direct mutation of Main state during preview. Suggested shape:

```js
{
  schemaVersion: 1,
  source: { type: "main", storageKey: "isf-rebuild-v1", importedAt },
  accounts: [{ id, name, role, sourceAccountId }],
  relationships: [{
    id,
    type, // income-deposit | auto-transfer | card-payment | utility-payment | loan-payment | savings-transfer | investment-transfer
    sourceAccountId,
    targetAccountId,
    label,
    amount,
    amountVisibility: "hidden-by-default",
    paymentDay: "",
    memo: "",
    confidence: "confirmed" | "recommended" | "needs-confirmation",
    sourceRef: { collection, id }
  }],
  candidates: [{ id, sourceRef, reason, recommended }]
}
```

Model recommendations:

- Generate account nodes from `inputs.accounts`, with fallback aliases only if Main data lacks accounts. [VERIFIED: current sanitizer behavior]
- Generate income-deposit relationships from `incomes[].allocations` when present; fall back to `income.accountId` when no allocation array exists. [VERIFIED: current sanitizer/calculator behavior]
- Generate savings/investment transfer relationships from `savingsItems[]` and `investItems[]` using `accountId`, item name, amount, maturity, and rate as detail metadata. [VERIFIED: current item fields]
- Generate automatic transfer relationships from `transfers[]`, preserving `label` and marking them confirmed/imported if they came from Main. [VERIFIED: current transfer schema]
- Generate fixed-expense candidates from `expenseItems[]` where `group` normalizes to fixed-expense paths or names match the locked recurring-payment keywords. Mark keyword hits as `recommended`, other fixed expenses as `needs-confirmation`. [VERIFIED: 10.8 context; current default groups/names]
- Exclude variable expenses and one-off spending from accepted relationships by default. Keep them only as reviewable candidates if the planner chooses a broader candidate review step. [VERIFIED: 10.8 context]
- Keep amount hidden in the map overview and reveal amount/payment day/memo in selection detail UI. [VERIFIED: 10.8 D-04/D-15]

Relationship type labels should be short and color-coded: income, transfer, card/payment, utility/subscription, loan/rental/insurance, savings, investment. Filter tabs are deferred; include type metadata now so later filters do not require migration. [VERIFIED: 10.8 D-05 and deferred ideas]

## UX/Implementation Risks

| Risk | Why It Matters | Planner Mitigation |
|---|---|---|
| Turning Account Map into a budget screen | Ordinary categories and one-off expenses would obscure real account relationships. | Import only income, savings, investment, transfers, and fixed-expense candidates by default; require explicit candidate acceptance. [VERIFIED: 10.8 D-01/D-02/D-11/D-12] |
| Showing too many amounts at once | User explicitly wants exact monthly amounts hidden by default. | Render relationship labels/types first; reveal amount/payment day/memo only on selected account/edge. [VERIFIED: 10.8 D-04/D-15] |
| Treating Main inferred fields as bank truth | Existing `accountId`, surplus routing, and default aliases are planning data, not live bank/card integrations. | Add `confidence` and `sourceRef`; distinguish confirmed manual transfers from inferred candidates. [VERIFIED: source read; requirements out-of-scope] |
| Breaking restored Main account-flow behavior | Phase 10.7 review fixes intentionally restored manual transfer UI and source-account network flow. | Keep Main changes to an entry card/mini-map and connector export; do not remove existing Main account/network code in this phase. [VERIFIED: 10.7-REVIEW-FIX.md] |
| XSS in relationship labels/imported data | Existing review fixed Portfolio attribute escaping; Main list renderers have prior hardening concerns. | Use DOM APIs/textContent/dataset for Account Map renderers; avoid `innerHTML` for user labels, account names, ids, and memo. [VERIFIED: 10.7 review; conventions docs] |
| Mobile first viewport crowding | Account Map must open with the map as primary visual and summary compact. | Summary bar should be small counts only; warnings/candidate review should live below or in a drawer. [VERIFIED: 10.8 D-14/D-16/D-17] |
| Over-building filters | Relationship-type filters are deferred. | Store `type`, but ship no filter tabs unless a later phase asks for focus mode. [VERIFIED: 10.8 deferred ideas] |
| Storage contract ambiguity | Account Map draft persistence is new and separate from Main. | Add an Account Map storage key and import-from-Main command; do not save accepted draft back into Main until an explicit future contract exists. [ASSUMED] |
