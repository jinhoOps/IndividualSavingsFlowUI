# Phase 10.8: Account Management Map - Context

**Gathered:** 2026-06-29
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 10.8 creates a dedicated Account Map experience for understanding real account-management relationships at a glance. This is not a budgeting surface and not another ordinary Step 1 financial editor. It focuses on real recurring account flows: automatic transfers, card payment account links, utility/telecom/insurance payments, savings transfers, and investment/brokerage transfers.

Main remains a simple entry point with a small mini-map or status card. The full account map, detailed relationship review, and relationship-first edits belong in a separate Account Map page.

</domain>

<decisions>
## Implementation Decisions

### Relationship Scope

- **D-01:** The first Account Map scope includes real recurring account relationships: automatic transfers, card/payment links, utility/telecom/insurance/loan/rental/subscription payments, savings transfers, and investment/brokerage transfers.
- **D-02:** Ordinary spending categories and one-off expenses are excluded from the primary map. They must not turn the Account Map into another budget screen.
- **D-03:** The default map shows high-level account-to-account flow first. Detailed card, utility, insurance, and payment links appear when the user selects an account or relationship.
- **D-04:** Exact monthly amounts are not shown everywhere by default. Selecting an account or relationship reveals monthly amount, payment day, memo, and related details.
- **D-05:** Relationship types are distinguished with colors and short labels. Relationship-type filter tabs are deferred as a later focus feature.

### Screen Location

- **D-06:** The primary Account Map screen is a separate page, expected to live under a dedicated route such as `apps/account-map/`.
- **D-07:** Main should expose only a lightweight mini-map or entry card that routes to the Account Map page. Main must not become the full account-management editor.
- **D-08:** Portfolio is not the destination for this feature. Account Map is its own destination because the user wants a focused map for account relationships, not a portfolio/stock allocation surface.

### Input and Draft Generation

- **D-09:** First use starts from a draft generated from Main data. Users should not face an empty account-map setup where they must create accounts first.
- **D-10:** After the draft is generated, editing is relationship-first: users add or modify relationships, and accounts are created or adjusted naturally through those relationships.
- **D-11:** The implementation should import Main income, savings, investment, and fixed-expense data into the draft. Variable expenses and one-off spending are excluded by default.
- **D-12:** Fixed expenses are initially shown as Account Map candidates. Users can exclude candidates before accepting the draft.
- **D-13:** If implementation cost is reasonable, obvious recurring-payment keywords such as `통신`, `전기`, `가스`, `수도`, `관리비`, `보험`, `대출`, `렌탈`, `구독`, and `카드` should be marked as recommended/likely candidates, while other fixed expenses remain confirmation-needed candidates. The minimum required behavior is still fixed-expense candidate listing.

### First Screen Layout

- **D-14:** The Account Map page should open with the full account-flow map as the primary visual. The map is the product's first-viewport signal.
- **D-15:** Selecting an account or relationship opens detail UI for linked cards, utility payments, savings/invest transfers, amounts, payment day, and memo.
- **D-16:** Web/desktop may show a small summary bar near the map with counts such as account count, relationship count, and confirmation-needed count.
- **D-17:** Mobile should keep the summary compact so it does not push the map out of view. Problems/warnings must not dominate the first screen.

### the agent's Discretion

- Decide the exact route name, Vite entry naming, and file split for the new Account Map app while following existing multi-page app conventions.
- Decide whether the first implementation uses keyword recommendation states or a simpler fixed-expense candidate checklist, as long as fixed expenses can be reviewed before import.
- Decide the exact graph layout and interaction mechanics, but preserve the product constraint that the map must remain readable at a glance before details are opened.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets

- `apps/main/modules/network-map-renderer.js` — Existing account network renderer is a useful reference for account-node/edge rendering, but it may need adaptation for a dedicated Account Map page.
- `apps/main/modules/sankey-builder.js` — Existing graph-data builder includes account/transfer concepts and can inform the Account Map data shape, but the new page should not make Main's primary Sankey complex again.
- `apps/main/modules/input-sanitizer.js` — Existing sanitized Main data is the source for draft generation. Import rules should use sanitized income, savings, investment, and fixed-expense items.
- `apps/main/modules/list-renderer.js` — Existing transfer list and balance hint logic are relevant for relationship labels and detail views.
- `src/core/storage/CompatibilityBridge.ts` and `src/core/storage/IsfStore.ts` — Storage bridge and active Step 1 snapshot access should be used rather than writing against old storage modules directly.
- `shared/components/app-header.js` — New Account Map route should integrate with shared navigation/status patterns.

### Established Patterns

- New routed app pages use an HTML shell under `apps/<route>/`, a Vite entry under `src/entries/`, page-specific modules/styles, and a Vite Rollup input in `vite.config.ts`.
- App-local JavaScript modules use kebab-case filenames and focused modules such as `*-controller.js`, `*-renderer.js`, and `*-builder.js`.
- Step 1 data changes must pass through sanitizer/persistence boundaries. The Account Map draft should consume Main data, not mutate Main directly during preview.
- User-editable data should be rendered safely with escaped text or DOM APIs, following recent Portfolio DOM hardening.

### Integration Points

- Main needs a small Account Map entry/mini-map card, not a full embedded editor.
- The dedicated Account Map page needs a Step 1 data connector that reads the latest sanitized Main snapshot and generates an editable draft.
- Vite, shared header navigation, and Playwright tests need updates if a new `apps/account-map/` route is added.
- Existing Phase 10.7 account-flow sidecar/handoff behavior should be reviewed so old saved account data can be used where helpful without reintroducing Step 1 complexity.

</code_context>

<specifics>
## Specific Ideas

- The user framed the feature as real account-management flow, not budget management.
- The user's primary examples were actual automatic transfer setup, credit/check card account links, and recurring utility/payment obligations.
- The final UX goal is a map where managed account relationships are easy to understand at a glance.
- The user prefers Main-data-based draft generation first, then relationship-first editing.
- The user explicitly wants a separate Account Map page plus a lightweight Main entry/mini-map, not a Portfolio-attached experience.
- The user wants future relationship filtering to become a later focus feature, not part of the initial 10.8 scope.

</specifics>

<deferred>
## Deferred Ideas

- Relationship-type filter tabs should become a later focus feature after the base map is usable.
- Live bank/card scraping remains out of scope; this phase should stay offline/static and user-managed.

</deferred>
